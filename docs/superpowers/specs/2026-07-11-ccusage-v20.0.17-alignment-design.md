# ccusage v20.0.17 对齐设计

## 背景与结论

本机曾安装已被上游撤回的 `ccusage 20.1.0`。该版本不执行当前官方版本使用的 Codex thread/fork replay 抑制，导致 TokenWatch 与命令行在 2026-07-10、2026-07-11 分别出现约 44% 和 41% 的 token 差额。

本机已通过 Homebrew 重新安装官方当前版本 `ccusage 20.0.17`。在同一份冻结 Codex 日志上，TokenWatch 当前解析器与 `ccusage 20.0.17 --offline` 的 2026-07-08 至 2026-07-11 input、cache read、output、reasoning 和 total 已逐字段一致。因此本轮不得删除或削弱现有 Codex replay、last-first、跨文件 first-wins、增量 checkpoint 与 continuity anchor 逻辑。

本轮把可复现兼容基线从：

- `ccusage v20.0.16 / e32cc4820df1e13f4399560e03f3858869738dc8`

更新为：

- `ccusage v20.0.17 / 88cdfa4fb201c92b163a34d0bbb097b68d3185cf`
- `--offline`
- 默认 Auto cost mode
- Asia/Shanghai 仅用于本次冻结日志的日期对账，不写死为产品时区

`v20.0.16 → v20.0.17` 的定价源码、`flake.lock` 中 LiteLLM/models.dev revision、内嵌 models.dev snapshot、fast multiplier overrides 均无差异。唯一产品相关的上游功能变化是 Claude advisor usage 计量。因此生产费率表和计价算法保持不变，只更新 parity 元数据与 fixture 名称。

## 目标

1. TokenWatch 对当前支持的 Codex rollout 在同一日志、时区和日期范围内输出与 ccusage v20.0.17 相同的 token 维度和总量。
2. TokenWatch 计入 Claude `message.usage.iterations` 中合法的 `advisor_message` usage，并按 advisor 自身模型单独计价。
3. TokenWatch 所有聚合总量遵守 ccusage 的非重复口径：`input + output + cache read + cache creation`；`reasoning` 保留为观察维度，但不再次加入 total。
4. 固定金额 fixture 明确记录 v20.0.17 commit，同时证明现有 21 个 offline Auto 金额期望保持不变。

## 非目标

- 不修改 Codex replay classifier、skip state、previous totals、loader dedup key 或增量缓存机制。
- 不修改 `litellm_prices.json`、`models-dev-pricing.json`、builtin price、long-context、fast、cache 1h/5m 或 Auto fallback 算法，因为 v20.0.17 与当前固定版本在这些文件上完全相同。
- 不增加运行时网络定价请求，也不让 App 调用外部 `ccusage` 二进制。
- 不扩张到 TokenWatch 当前未支持的 ccusage 数据源或额外目录发现规则。
- 不在本轮重做 Dashboard 的 app-specific 成本比例分摊或 cache hit rate 展示；本轮只修 token total 的重复计数。
- 不重写已经执行完毕的历史计划；必要时仅增加指向本设计的 superseding amendment。

## 方案比较

### 方案 A：按 v20.0.17 精确增量对齐（采用）

保留已经与 v20.0.17 对齐的 Codex 和价格实现，只实现 advisor 展开、统一非重复 total，并升级 parity 元数据。该方案改动与上游实际差异一一对应，风险和验证范围最小。

### 方案 B：只重装 CLI 并更新版本元数据

该方案可以消除 Codex 的大差额，但仍漏算 Claude advisor usage，Dashboard 和 Sessions 仍会把 reasoning 重复加入 total，不能满足完整对齐。

### 方案 C：整体移植 v20.0.17 provider adapter

该方案会把 TokenWatch 未支持的数据源、目录发现和 CLI 配置语义一并带入，产生大量无关变更。当前差异只有一个 Claude 功能点，没有整体移植的收益。

## 组件设计

### 1. 兼容基线与定价契约

将定价 parity fixture 从 `ccusage-v20.0.16.json` 重命名为 `ccusage-v20.0.17.json`，并把：

- `ccusageVersion` 更新为 `v20.0.17`；
- `ccusageCommit` 更新为 `88cdfa4fb201c92b163a34d0bbb097b68d3185cf`；
- suite 名称、fixture 路径和版本断言同步更新。

以下值保持不变，并继续在测试中逐项断言：

- LiteLLM revision 与 source SHA-256；
- models.dev source SHA-256；
- fast overrides SHA-256；
- auto-review fallback SHA-256；
- 21 个 fixture case 的输入和 `expectedUSD`。

这不是仅替换标签：测试同时锁定 v20.0.17 commit 和全部资源哈希，防止未来在版本号不变时误换价格数据。

### 2. Claude advisor usage 展开

主 Claude billing 行继续使用现有 `ClaudeUsageLine` / `ClaudeNormalizedUsageRecord` 解码和校验，不改变 direct、AgentProgress、optional ID、null guard、timestamp 或 sidechain 规则。

新增独立的窄 advisor envelope 解码器，只读取顶层：

```text
message.usage.iterations[]
```

advisor envelope 必须先整体通过 schema 解码；任一 iteration 的 token 字段类型非法会使本行不展开任何 advisor，但不影响已经解析成功的主记录。解码成功后，每个 iteration 仅在同时满足以下条件时生成 advisor usage：

- `type == "advisor_message"`；
- `model` 存在且非空；
- token 字段能按现有 Claude 非负整数规则解码。

advisor 解码必须与主记录解码隔离。原始行通过既有 compact usage marker/null prefilter 后，advisor envelope 的普通坏类型仅忽略 advisors，已经合法的主 usage 仍保留，这与 ccusage v20.0.17 对同一行先解析主记录、再独立解析 advisors 的行为一致。既有 raw prefilter 仍逐字扫描整行；iteration 内出现其 unsupported compact `:null` 时，上游 20.0.17 也会跳过整行，因此本轮不把该 guard 改成 scope-aware。

一条合法主记录展开如下：

| 字段 | 主记录 | advisor 记录 |
|---|---|---|
| timestamp/session/request/sidechain/cwd | 原值 | 继承原值 |
| model | 主模型 | iteration 自身模型 |
| usage | 顶层 usage | iteration token 字段 |
| upstream cost | 顶层 `costUSD` | `nil`，由 advisor 模型本地计价 |
| message ID | 原 ID 或本地稳定 ID | `<主 ID>:advisor:<index>` |
| record UUID | 原 UUID 或本地稳定 UUID | `<主 UUID>:advisor:<index>` |

非 advisor iteration 不产生额外 entry。advisor usage 不从主 usage 中扣除；上游明确把顶层 usage 归于主模型，再把 advisor usage 作为额外 billing row。

现有 parser 的单行返回值从 optional candidate 调整为 candidate 数组。全量、增量 committed 行和 provisional tail 必须使用同一展开函数，避免 fresh/incremental 结果漂移。advisor 合成 message ID 继续进入现有 Claude exact/sidechain 单遍去重器，不新增第二套去重实现。

### 3. 非重复 token total

在 `TokenUsage` 增加共享派生属性，使用饱和加法计算：

```text
inputTokens
+ outputTokens
+ cacheReadInputTokens
+ totalCacheCreationTokens
```

该属性不加入 `reasoningTokens`。理由是：

- Codex `output_tokens` 已包含 `reasoning_output_tokens`；
- OpenCode adapter 已把 source total 缺口并入 output，reasoning 仅作展示；
- Claude 当前 reasoning 为零；
- ccusage 的 daily/session total 同样不把 reasoning 作为额外可加维度。

`UsageSummaryAccumulator` 和 `RecentSessionUsageAccumulator` 都累加这一共享属性，不再各自手写包含 reasoning 的数组公式。reasoning 的 overall/byDay/byModel/session 数值仍独立聚合并交给 UI 展示。

这样 Overview、趋势、工具/模型/项目份额以及 Sessions 页会得到同一 total 口径；状态栏已有的非重复公式保持不变。

## 错误处理与缓存

- malformed 主记录仍按现有规则整行跳过。
- 通过既有 raw prefilter 后，malformed advisor envelope 或任一 iteration 的普通非法 token 类型会阻止本行全部 advisor 展开，但不影响合法主记录；unsupported compact `:null` 仍按 pinned raw guard 跳过整行。
- envelope 中所有 iterations 先整体通过 flattened token schema；随后未知 iteration 类型和空 model 只过滤对应 iteration，不影响同一 envelope 中其他合法 advisor。
- advisor 没有 source `costUSD`，Auto mode 必须按 advisor model 查本地价格；未知模型沿用现有 `$0` 与一次性 miss 日志。
- Claude 文件缓存结构仍保存 `[ParsedUsageEntry]`，不需要磁盘迁移。进程升级后内存缓存自然清空。
- 增量 append 的 committed offset、continuity anchor 和 provisional tail 规则不变；同一行的所有展开 entries 必须作为一个不可分割的解析结果加入候选数组。
- pricing lookup cache 无 schema 或 scope 变化。

## 测试设计

严格执行 RED → GREEN：每个行为测试必须先在当前生产实现上因缺失目标行为而失败，再写最小生产修改。

### Claude advisor

1. direct 主记录包含一个 `advisor_message`：得到主记录和 advisor 两条 entry。
2. advisor 使用自身 model/usage，继承 timestamp/session/request/sidechain/cwd，且 `upstreamCost == nil`。
3. 主记录保留显式 `costUSD`，advisor 由自身模型计算成本。
4. 多个 iterations 中仅合法 advisor 生成 entry；普通 iteration 与空 model 被忽略。
5. advisor envelope 损坏时主记录仍保留。
6. advisor 合成 ID 在重复文件和 sidechain 流程中保持稳定去重。
7. advisor 行经历 provisional → committed → append 后，增量结果与 fresh full scan 深度一致。

### Token total

1. reasoning 仍聚合到 reasoning 维度，但 overall/byDay/byModel total 不包含它。
2. `input=1, output=1, reasoning=Int.max` 的 total 为 2，证明极值不会用饱和掩盖重复计数。
3. Codex parser→aggregator fixture 保留 output/reasoning 原值，但 total 等于 input + cache + output。
4. OpenCode source-total fallback 保留 reasoning 展示，同时 total 不重复加入 reasoning。
5. Recent Sessions 的 row、model breakdown 和 summary 使用同一非重复 total。
6. Claude reasoning 为零的既有金额和 token 断言不变。

### v20.0.17 parity

1. fixture 元数据断言 version、commit、offline、Auto 和全部资源 SHA。
2. 21 个固定金额 case 全部以 `1e-9` 容差通过。
3. 对 v20.0.17 advisor fixture 同时比较 token、model 和 cost。
4. 用冻结 Codex 日志比较 App 与本机 ccusage v20.0.17 的每日 input/cache/output/reasoning/total。
5. 用同一冻结日志比较 App 聚合成本与 `ccusage codex daily --offline` 的每日 `costUSD`。

## 验收标准

- `/opt/homebrew/bin/ccusage --version` 输出 `ccusage 20.0.17`。
- Codex 冻结日志四天逐字段 token diff 为空。
- Claude advisor 合成 fixture 在 App 与 ccusage v20.0.17 中得到相同 billing rows 和金额。
- Dashboard、各聚合维度和 Sessions total 均不重复加入 reasoning。
- v20.0.17 的 21 个 pricing parity case 全部通过，资源哈希保持固定。
- 相关定向 suites、完整 `TokenWatchTests` unit target、Debug build 均成功；没有以 `build-for-testing` 代替真实测试执行。
- 生产源码不包含临时诊断路径、冻结日志路径或对本机外部二进制的运行时依赖。
